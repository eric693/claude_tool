技術選型

為什麼選這個組合？

這不是隨機選的。每個技術都有明確的選用理由，也有明確的替代方案說明。

Next.js 15 App Router
Server Components 讓 AI API 呼叫在伺服器端執行，API key 不洩漏到前端。Edge Runtime 讓 Streaming 延遲降到最低。Server Actions 讓你省掉寫 API route 的樣板代碼。
Supabase
PostgreSQL + Auth + Realtime + Storage + pgvector（後面 RAG 課用到）一包解決。免費額度夠整個課程用。最重要的是：Row Level Security 讓多用戶資料隔離不需要額外寫邏輯。
TypeScript
Claude API 回傳的結構複雜，Tool Use 的 JSON schema 更複雜。沒有型別保護，在第 5 堂課 RAG 的時候你會崩潰。TypeScript 是這個課程的隱性安全網。
Vercel
零設定 Streaming 支援。Edge Functions 讓 AI API 呼叫的冷啟動幾乎不存在。免費方案有 100GB 頻寬，足夠課程練習和小型上線。
Claude Code 工作流

用 AI 建造 AI 應用

Claude Code 是你在終端機裡的工程師夥伴。它能讀整個 codebase、寫程式、執行指令、除錯。但大多數人用法錯誤 — 他們把它當 ChatGPT，一次問一個問題。正確用法是給它任務、給它上下文、讓它自己跑。

Claude Code 安裝：npm install -g @anthropic-ai/claude-code，然後在專案根目錄執行 claude 即可。需要 Anthropic API key。

Prompt 解剖：如何讓 Claude Code 真正懂你

一個有效的 Claude Code prompt 由五個層次構成。少了任何一層，輸出品質就會明顯下降。

PROMPT ANATOMY — 五層結構
ROLE
你是什麼 — 定義 Claude 的身份和專業背景
範例：「你是一個精通 Next.js App Router 和 Supabase 的全端工程師」
CONTEXT
現在在哪 — 專案狀態、已存在的檔案、技術限制
範例：「這是一個 Next.js 15 專案，使用 App Router，src/ 目錄結構，已安裝 @anthropic-ai/sdk 和 @supabase/ssr」
TASK
做什麼 — 具體、可驗證的任務描述
範例：「在 src/app/api/chat/route.ts 建立一個 POST endpoint，接收 messages 陣列，呼叫 Claude API 並以 SSE 串流回傳」
FORMAT
輸出長什麼樣 — 期望的輸出格式和結構
範例：「直接輸出可執行的 TypeScript 程式碼，不需要解釋，包含所有 import」
CONSTRAINT
不能做什麼 — 明確的限制和邊界
範例：「不使用 any 型別，不使用 try-catch 包整個函式，錯誤要精確 throw 到對應的地方」

實際範例對比

BAD PROMPT

「幫我寫一個聊天功能」

問題：沒有 role，沒有 context，task 模糊，不知道要產出什麼，Claude 會自己假設一堆東西然後全都猜錯。

GOOD PROMPT

「你是精通 Next.js 15 App Router 的工程師。這個專案用 @anthropic-ai/sdk 和 Supabase。在 src/app/api/chat/route.ts 建立 POST handler，接收 { messages: Message[], system?: string }，用 claude-sonnet-4-20250514，以 SSE 格式串流回傳，text delta 包在 data: {"text":"..."}\n\n 格式裡，結束送 data: [DONE]。只輸出程式碼。」

給 Claude Code 的常用指令模式

IMPLEMENT
「實作 [功能]，在 [檔案路徑]，使用 [技術]，符合 [現有架構]。只輸出程式碼，不需要解釋。」
FIX
「這個錯誤：[貼上錯誤訊息]。相關檔案：[路徑]。找出根本原因並修正。不要改動無關的程式碼。」
REFACTOR
「把 [檔案] 裡的 [函式/模組] 重構成 [目標結構]。保持所有現有測試通過。不改變對外 API。」
EXPLAIN
「解釋 [程式碼/概念] 如何運作，假設我已知 [已知知識]，重點放在 [我不確定的部分]。」
CLAUDE.md 設定

讓 Claude Code 永遠記得你的專案

CLAUDE.md 是放在專案根目錄的設定檔。Claude Code 每次啟動都會自動讀取它，讓你不需要每次重新解釋專案背景、技術限制、命名規範。

這是省下最多時間的單一最大優化。好的 CLAUDE.md 讓你節省的解釋時間累積下來是幾十小時。

MARKDOWN CLAUDE.md（專案根目錄） 
# 專案背景
這是一個 AI SaaS 應用，使用 Next.js 15 App Router + TypeScript + Supabase + Claude API。
目標：讓用戶可以登入並使用各種 AI 功能（對話、文件分析、行程規劃等）。

# 技術棧
- Framework: Next.js 15，App Router，src/ 目錄
- Language: TypeScript，strict mode
- Database: Supabase（PostgreSQL + Auth + Realtime）
- AI: @anthropic-ai/sdk，model = claude-sonnet-4-20250514
- Styling: Tailwind CSS v4
- Deploy: Vercel

# 目錄結構
src/app/api/         - API routes（所有後端邏輯）
src/app/(auth)/      - 登入/登出頁面（公開路由）
src/app/(app)/       - 登入後的頁面（保護路由）
src/components/      - React 元件
src/lib/claude.ts    - Claude API 封裝
src/lib/supabase/    - Supabase client 工具
src/types/index.ts   - 共用型別

# 程式碼規範
- 不使用 any 型別，用 unknown 然後 narrow
- API route 一律加 input validation（用 zod）
- 永遠不在前端直接呼叫 Claude API，全部走 /api/
- Supabase 操作：前端用 anon key，後端管理用 service role
- 函式命名：動詞開頭（getUser, createConversation, streamChat）
- 元件命名：PascalCase，檔案名稱與 export 名稱一致

# 環境變數
ANTHROPIC_API_KEY              - Claude API key（只在 server side 用）
NEXT_PUBLIC_SUPABASE_URL        - Supabase project URL
NEXT_PUBLIC_SUPABASE_ANON_KEY   - 前端用
SUPABASE_SERVICE_ROLE_KEY       - 後端管理用（絕對不給前端）

# 重要限制
- 不修改 app/layout.tsx 除非有明確指示
- 新增 API route 時必須加 rate limiting
- 所有資料庫操作必須確認 RLS policy 存在

System Prompt 設計

為你的 AI 應用設計 System Prompt

System Prompt 是每次呼叫 Claude API 時傳入的背景指令。它決定了你的 AI 應用的個性、能力邊界和回覆風格。這不是「調整一下 AI 語氣」那麼簡單的事，它是你的產品差異化的核心。

旅遊規劃平台的 System Prompt 範例

TYPESCRIPT src/lib/prompts/travel.ts 
export const TRAVEL_PLANNER_SYSTEM = `
你是一個專業的旅遊規劃師，擁有全球地理知識和旅行實務經驗。

## 你的能力
- 根據用戶的預算、時間、偏好生成完整行程
- 提供具體的景點、餐廳推薦（包含評分和特色）
- 估算交通時間和費用
- 識別潛在問題（旺季、簽證、天氣）

## 回覆規則
- 行程以天為單位，每天上午/下午/晚間分段
- 費用估算以台幣表示，標注匯率假設
- 景點推薦必須包含：名稱、預計停留時間、建議理由
- 不確定的資訊明確標注「建議出發前再確認」

## 工具使用
當用戶詢問即時資訊（天氣、航班、票價），你會呼叫對應的工具取得最新數據，
不要回答你不確定的即時資訊。

## 限制
不提供醫療建議，不推薦非法活動，不對政治敏感地區表態。
`

// 動態注入用戶偏好
export function getTravelSystemPrompt(userPrefs?: {
  budget?: 'backpacker' | 'mid-range' | 'luxury'
  travelStyle?: 'cultural' | 'adventure' | 'relaxation'
}) {
  let prompt = TRAVEL_PLANNER_SYSTEM
  if (userPrefs?.budget) {
    prompt += `\n\n用戶偏好預算等級：${userPrefs.budget}，推薦請符合此預算範圍。`
  }
  if (userPrefs?.travelStyle) {
    prompt += `\n用戶旅遊風格：${userPrefs.travelStyle}，優先推薦符合此風格的體驗。`
  }
  return prompt
}

System Prompt 設計原則

明確能力邊界
列出 Claude「能做什麼」和「不能做什麼」。模糊的邊界讓 Claude 在邊緣案例時猜測，而猜測結果不可預期。
具體的輸出格式
如果你的前端需要解析特定 JSON 結構，在 system prompt 裡定義它。不要依賴「Claude 大概會這樣輸出」。
動態注入用戶上下文
system prompt 不是靜態的。每次呼叫時注入用戶設定、歷史偏好、當前狀態。這讓每個用戶感覺 AI 記得他。
Tool Use 說明
如果這個 session 有提供工具，在 system prompt 裡說明何時該呼叫工具。Claude 需要知道工具的使用時機，不是只知道工具存在。

API Route 實作

Claude API 封裝 + Route Handler

TYPESCRIPT src/lib/claude.ts 
import Anthropic from '@anthropic-ai/sdk'
import type { Message } from '@/types'

// 單例模式，避免每次 request 都重新初始化
const anthropic = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY!,
})

export const DEFAULT_MODEL = 'claude-sonnet-4-20250514'
export const DEFAULT_MAX_TOKENS = 4096

// 非串流呼叫（用於後台任務、摘要生成等）
export async function askClaude(
  messages: Message[],
  system?: string,
  maxTokens = DEFAULT_MAX_TOKENS
): Promise<string> {
  const res = await anthropic.messages.create({
    model: DEFAULT_MODEL,
    max_tokens: maxTokens,
    system,
    messages,
  })
  const block = res.content[0]
  if (block.type !== 'text') throw new Error('Unexpected response type')
  return block.text
}

// 串流呼叫（用於聊天介面，SSE 格式）
export async function streamClaude(
  messages: Message[],
  system?: string,
  maxTokens = DEFAULT_MAX_TOKENS
) {
  return anthropic.messages.stream({
    model: DEFAULT_MODEL,
    max_tokens: maxTokens,
    system,
    messages,
  })
}

// Tool Use 呼叫（用於旅遊、叫車等需要外部 API 的場景）
export async function claudeWithTools(
  messages: Message[],
  tools: Anthropic.Messages.Tool[],
  system?: string
) {
  return anthropic.messages.create({
    model: DEFAULT_MODEL,
    max_tokens: 4096,
    system,
    tools,
    messages,
  })
}
TYPESCRIPT src/app/api/chat/route.ts 
import { NextRequest, NextResponse } from 'next/server'
import { z } from 'zod'
import { streamClaude } from '@/lib/claude'
import { createClient } from '@/lib/supabase/server'

// Input validation schema
const ChatSchema = z.object({
  messages: z.array(z.object({
    role: z.enum(['user', 'assistant']),
    content: z.string().min(1).max(10000),
  })).min(1).max(50),
  conversationId: z.string().uuid().optional(),
  system: z.string().max(5000).optional(),
})

export async function POST(req: NextRequest) {
  // 1. 驗證用戶身份
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }

  // 2. 驗證 input
  const body = await req.json()
  const parsed = ChatSchema.safeParse(body)
  if (!parsed.success) {
    return NextResponse.json(
      { error: 'Invalid input', details: parsed.error.flatten() },
      { status: 400 }
    )
  }

  const { messages, system } = parsed.data

  // 3. 建立 SSE 串流
  const encoder = new TextEncoder()
  const stream = new ReadableStream({
    async start(controller) {
      let fullResponse = ''
      try {
        const claudeStream = await streamClaude(messages, system)

        for await (const chunk of claudeStream) {
          if (
            chunk.type === 'content_block_delta' &&
            chunk.delta.type === 'text_delta'
          ) {
            const text = chunk.delta.text
            fullResponse += text
            controller.enqueue(
              encoder.encode(`data: ${JSON.stringify({ text })}\n\n`)
            )
          }
        }

        // 4. 串流結束後，把訊息存到 Supabase
        if (parsed.data.conversationId) {
          await supabase.from('messages').insert([
            { conversation_id: parsed.data.conversationId, role: 'user', content: messages.at(-1)!.content },
            { conversation_id: parsed.data.conversationId, role: 'assistant', content: fullResponse },
          ])
        }

        controller.enqueue(encoder.encode('data: [DONE]\n\n'))
      } catch (err) {
        const msg = err instanceof Error ? err.message : 'Internal error'
        controller.enqueue(
          encoder.encode(`data: ${JSON.stringify({ error: msg })}\n\n`)
        )
      } finally {
        controller.close()
      }
    }
  })

  return new Response(stream, {
    headers: {
      'Content-Type': 'text/event-stream',
      'Cache-Control': 'no-cache, no-transform',
      'Connection': 'keep-alive',
      'X-Accel-Buffering': 'no',
    },
  })
}
Streaming 串流

前端接收串流 + React Hook

把串流邏輯封裝成 custom hook，讓 ChatInput 和 ChatWindow 都能乾淨地使用，不需要在元件裡寫 ReadableStream 邏輯。

TYPESCRIPT src/hooks/useChat.ts 
import { useState, useCallback } from 'react'
import type { Message } from '@/types'

export function useChat(system?: string) {
  const [messages, setMessages] = useState<Message[]>([])
  const [streaming, setStreaming] = useState(false)
  const [currentText, setCurrentText] = useState('')

  const sendMessage = useCallback(async (content: string) => {
    const userMsg: Message = { role: 'user', content }
    const nextMessages = [...messages, userMsg]
    setMessages(nextMessages)
    setStreaming(true)
    setCurrentText('')

    try {
      const res = await fetch('/api/chat', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ messages: nextMessages, system }),
      })

      if (!res.ok) throw new Error(await res.text())

      const reader = res.body!.getReader()
      const decoder = new TextDecoder()
      let accumulated = ''

      while (true) {
        const { done, value } = await reader.read()
        if (done) break

        const lines = decoder.decode(value, { stream: true }).split('\n')
        for (const line of lines) {
          if (!line.startsWith('data: ')) continue
          const raw = line.slice(6)
          if (raw === '[DONE]') break
          const { text } = JSON.parse(raw) as { text?: string }
          if (text) {
            accumulated += text
            setCurrentText(accumulated)
          }
        }
      }

      // 串流完成：把 assistant 訊息存進 messages
      setMessages(prev => [
        ...prev,
        { role: 'assistant', content: accumulated }
      ])
    } finally {
      setStreaming(false)
      setCurrentText('')
    }
  }, [messages, system])

  return { messages, streaming, currentText, sendMessage }
}
Auth + 資料庫

Supabase Auth + Row Level Security

正確的 RLS 設計讓資料隔離變成資料庫層的保障，不依賴應用程式邏輯。這是 AI 多用戶應用的基本安全要求。

SQL supabase/migrations/001_init.sql 
-- ===== PROFILES =====
create table profiles (
  id           uuid references auth.users on delete cascade primary key,
  display_name text,
  email        text unique not null,
  created_at   timestamptz default now() not null
);

-- 新用戶自動建立 profile
create or replace function handle_new_user()
returns trigger as $$
begin
  insert into public.profiles(id, email)
  values(new.id, new.email);
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure handle_new_user();

-- ===== CONVERSATIONS =====
create table conversations (
  id         uuid default gen_random_uuid() primary key,
  user_id    uuid references profiles(id) on delete cascade not null,
  title      text not null default '新對話',
  created_at timestamptz default now() not null,
  updated_at timestamptz default now() not null
);

create index idx_conversations_user_id on conversations(user_id);

-- ===== MESSAGES =====
create table messages (
  id              uuid default gen_random_uuid() primary key,
  conversation_id uuid references conversations(id) on delete cascade not null,
  role            text check(role in('user', 'assistant')) not null,
  content         text not null,
  created_at      timestamptz default now() not null
);

create index idx_messages_conversation_id on messages(conversation_id);

-- ===== ROW LEVEL SECURITY =====
alter table profiles     enable row level security;
alter table conversations enable row level security;
alter table messages     enable row level security;

-- Profiles: 只能讀/寫自己
create policy "profiles: self only" on profiles
  for all using (auth.uid() = id);

-- Conversations: 只能操作自己的
create policy "conversations: own" on conversations
  for all using (auth.uid() = user_id);

-- Messages: 只能操作屬於自己 conversation 的
create policy "messages: own conversations" on messages
  for all using (
    conversation_id in (
      select id from conversations where user_id = auth.uid()
    )
  );

RLS policy 設定完後，用 Supabase Dashboard 的 SQL Editor 執行 select * from conversations，應該要回傳 0 筆資料（因為你沒有用 authenticated 身份查詢）。這是確認 RLS 正確運作的最快方式。

import { createServerClient } from '@supabase/ssr'
import { cookies } from 'next/headers'

export async function createClient() {
  const cookieStore = await cookies()
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll() },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value, options }) =>
            cookieStore.set(name, value, options)
          )
        },
      },
    }
  )
}
